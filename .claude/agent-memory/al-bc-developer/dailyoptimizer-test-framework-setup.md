**Correct app IDs for native BC `[Test]` codeunit dependencies** (added to app.json 2026-06-16, no test app/codeunit existed before this). The commonly-circulated GUID list for these has wrong IDs for two of the four — verified against the live Microsoft_*.app NavxManifest.xml after download:

- Library Assert: `dd0be2ea-f733-4d65-bb34-a28f4624fb14` — correct as commonly documented.
- Library Variable Storage: `5095f467-0a01-4b99-99d1-9ff1237d286f` — NOT `e7320ebb-08b3-4e1e-8c4b-37ae5bb1f994` (that GUID does not resolve to anything in this BC version's NuGet feed).
- Any: `e7320ebb-08b3-4406-b1ec-b4927d3e280b` — NOT `5d86850b-0d76-4eca-bd7b-951ad998e997` (that GUID belongs to a different app, not "Any").
- System Application Test Library: `9856ae4f-d1a7-46ef-89bb-6ef056398228` — correct as commonly documented, but as of 2026-06-16 the live NL_Test tenant does not serve this one at all (`al_downloadsymbols` against the sandbox returns 5 of 6); had to pull it from `globalSourcesOnly=true` instead, where it resolved at an older 26.5.x version while everything else resolved at 28.1.x. Harmless since no test code directly references its codeunits.

**AL codeunit name vs app/package name mismatch**: the codeunit inside the "Library Variable Storage" app is actually named `Library - Variable Storage` (with a hyphen, BC's "Library - X" naming convention), not "Library Variable Storage". Declare the variable as `Codeunit "Library - Variable Storage"`, not `Codeunit "Library Variable Storage"` — the latter compiles to AL0185 "is missing" even though the dependency app downloaded fine. Verified by inspecting `SymbolReference.json` inside the .app (it's a zip).

**Live tenant version drift**: NL_Test sandbox (tenant a60762e1-df10-4e4b-8f44-174c51589110) is actually running BC platform 28.1.x / runtime 17.0 as of 2026-06-16, even though app.json still declares `"application": "26.0.0.0"` and `"runtime": "15.0"`. Symbol downloads must target the live sandbox (or globalSourcesOnly without pinning) to get a consistent 28.1.x set — mixing cached 26.5.x and freshly-downloaded 28.x/28.1.x packages in `.alpackages` causes ambiguous-reference-style compile errors (e.g. "Codeunit X is missing" even though an .app with that name exists, because two different versions of the same app coexist in the folder). Fix: `rm -rf .alpackages/*` then re-download from a single consistent source before debugging "missing symbol" errors that don't make sense.

**Why:** Burned ~30 min in the 2026-06-16 session chasing phantom "missing codeunit" errors that were actually wrong GUIDs + stale/mixed-version symbol cache, not real AL mistakes.
**How to apply:** If a future session needs to add more `[Test]` codeunits or touch app.json dependencies, reuse the verified GUIDs above directly rather than re-deriving them, and always do a clean `.alpackages` wipe + single-source download when symbol errors look nonsensical. See [[project-dailyoptimizer]] for the test codeunit IDs already in use (60020/60021).

**al_publish / al_run_tests MCP tool unreliability observed 2026-06-16**: `al_publish` against NL_Test (tenant a60762e1-df10-4e4b-8f44-174c51589110) returned `succeeded:false` with a generic `PublishFailed` error and zero diagnostic detail on every single attempt (varying schemaUpdateMode, forceUpgrade, noCache, appPath vs projectPath, version bumps 28.0.0.0/.1/.2) — yet `al_run_tests` against codeunit 60020 ("Day Planning Creation Tests", newly created that session, never existed before) returned a recognized result (`1 skipped`, not "object not found"), proving the publish actually succeeded server-side at least once despite every tool call reporting failure. `al_run_tests` then consistently reported `0 passed, 0 failed, 1 skipped (0ms)` for every codeunit/method/company combination tried, with no error text — never actually executing the test body. Root cause undetermined (likely a company-context or test-run resolution issue independent of the AL code, since this happened identically for both a "happy path" test and the whole-suite test runner codeunit 60021). Not fixed by: rebuilding, version-bumping app.json, re-authenticating, or specifying `company`.
**How to apply:** If `al_publish` reports failure but the object clearly exists server-side (check via `al_run_tests` returning a non-"not found" result for a newly-added object ID), trust the server state over the tool's reported success flag. If `al_run_tests` returns `skipped` with 0ms and no detail for every method tried, this is very likely a tool/environment-level test-run resolution problem, not a bug in the test codeunit — don't waste time rewriting test logic in response to silent skips. Recommend falling back to opening Test Tool page 130400 directly in the BC web client to actually execute and see real pass/fail/error output when this happens.

**Update (2026-07-16): both tools worked cleanly in a later session** — `al_publish` (with `skipBuild=true` against an already-built `.app`) returned a real success, and `al_run_tests` against codeunit 60020 (passing `environmentName='NL_Test'`, `environmentType='Sandbox'`, `tenant='a60762e1-df10-4e4b-8f44-174c51589110'` explicitly — `al_run_tests` with NO connection args at all fails fast with `ConnectionFailed`/"No server connection is configured", it does not fall back to `launch.json` on its own) returned full real per-method PASS/FAIL detail, not a skip. So the unreliability documented above is NOT universal/permanent — always attempt a real `al_run_tests` call (with explicit environment/tenant params pulled from `.vscode/launch.json`) before assuming it'll silently skip; only fall back to "trust server state"/Test Tool page 130400 if this attempt actually reproduces the skip/failure pattern again.
**Correction, same day, minutes later:** a second independent `al_run_tests` call against the exact same codeunit 60020 with the exact same explicit `environmentName`/`environmentType`/`tenant` params (no code changes in between) reproduced the OLD `0 passed, 0 failed, 1 skipped (0ms)` pattern again. So this is genuinely intermittent/flaky even with correct params, not something that got "fixed" — two concurrent agent runs got two different outcomes back to back. Don't conclude from one clean run that the tool is now reliable; treat each `al_run_tests` result on its own merits, and if it comes back as a 0ms skip, fall back to Test Tool page 130400 for that specific verification rather than assuming code is broken or re-trying indefinitely.
