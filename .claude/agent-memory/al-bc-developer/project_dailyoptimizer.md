**Pool Resource Scheduler pagination (added 2026-07-08):** Page 50600 "DHX Scheduler (Pool Resource)" only ever renders one resource page at a time (default 200) instead of the whole company (~34,725 resources) because DHTMLX Scheduler has no server-side pagination hook. Pattern: `codeunit_50604_DHXDataHandler.al` has `GetYUnitElementsJSON_Pool` (now takes `var PageSkip`, `PageSize`, `var TotalResourceCount`) and a new `local procedure CollectPageResourceNos_Pool` that walks the SAME ResGroup‚ÜíPool‚ÜíResource nested loop as the tree-builder purely to count leaves and build a page-window `Dictionary of [Code[20], Boolean]` ‚Äî this doubles the enumeration cost of that nested loop (acceptable; it's not proportional to full JSON-building cost) but only when the requested page turns out to be out of range does it re-run a second time to rebuild the dictionary against the clamped skip. A separate public `FindResourcePageIndex` reruns the same walk to resolve a searched resource's 0-based position for the page's search box. The events query (query 50604 `Capacity Per Day Per Resource`) is bounded to the page via its previously-unused `Resource_No__Filter` filter element, OR-joined (`'RES1|RES2|...'`) ‚Äî confirmed safe because `Query.SetFilter`'s filter-expression parameter is always unbounded `Text` regardless of the underlying field's type (`Code[20]`), as long as the filter-building local variable is also declared as plain `Text` (not `Text[N]`).
**How to apply:** If the Resource-variant scheduler (`GetYUnitElementsJSON_Resource`) ever needs the same treatment, mirror this pattern rather than sharing code with the Pool variant ‚Äî the brief explicitly kept them separate (`GetUniqueResGroupFromCapacity`'s shared signature was NOT touched).

**Embedded resource files (added 2026-07-08):** Bundling a static file (e.g. CSV lookup data) into the `.app` requires declaring `"resourceFolders": ["src/data"]` in `app.json` ‚Äî this is a BC 2024 wave 2 / update 25.2+ feature and is NOT automatic; plain loose files anywhere in the project (project root, an undeclared folder, etc.) are silently dropped from the package even though `al_build` reports success. Verified by actually unzipping the built `.app` (it's a ZIP with a ~40-byte header before the `PK\x03\x04` signature) and checking for a `PackagedResources.json` entry plus `resources/<file>` entries. `NavApp.GetResourceAsText(ResourceName, TextEncoding::UTF8)` (and `GetResourceAsStream`/`GetResourceAsJson`) take `ResourceName` **relative to the declared folder**, not the project-relative path ‚Äî e.g. `src/data/FirstNames.csv` on disk is read back as just `'FirstNames.csv'`. `codeunit_50602_CreateDemoData.al`'s `GetNextName()` uses these CSVs (635 first names x 1000 last names = 635,000 combos) with a collision-free permutation ‚Äî `PermutedIndex := (Counter * 392453) mod 635000` computed in `BigInteger` (the raw `Counter * Multiplier` product overflows 32-bit `Integer` well before the ~34,725 resources created per demo-data run) ‚Äî 392453 was chosen because it's coprime with 635000 (`= 2^3 * 5^4 * 127`), making the mapping a bijection over `[0, 635000)` so every drawn name pair is unique as long as draws stay under 635,000.
**Why:** User showed a screenshot of many demo Resources all named "Jennifer Martinez" ‚Äî the old `GetRandomName()` drew from only 20x20=400 combos via `Random()`, guaranteeing duplicates at ~34,725 draws.

**Bug fix (2026-07-09): blank "Daily Optimizer Setup" singleton, superseded same day by "always force" (see below).** `page_50654_DailyOptimizerSetup.al`'s `OnOpenPage` auto-inserts a BLANK singleton record the first time anyone opens that page (`if not Rec.Get() then Rec.Insert()`), independent of whether Create Demo Data has run. `CreateDailyOptimizerSetupDefault()` used to whole-record-exit via `if Setup.Get() then exit;`, so if the page had already been opened once, the blank record "existed" and the procedure silently skipped populating every field ‚Äî including `Base Calendar`, which `codeunit_50610`'s `TestField("Base Calendar")` depends on downstream. First fix attempt switched to per-field blank checks (`if Setup."Base Calendar" = '' then ...`); also: the demo Base Calendar code changed from reusing generic `'BASIS'` to a dedicated `Code = 'DEMOCAL'` / `Name = 'DEMOPLANNING'` (Code is `Code[10]`, too short for "DEMOPLANNING" (12 chars), so the full label lives in Name instead) via new `CreateDemoCalendar()` (replaces the now-deleted `EnsureBaseCalendar()`), which ‚Äî unlike the other `Ensure*` helpers in this codeunit that are permanent/unlogged ‚Äî IS `LogRecord`'d and recreated fresh every run, avoiding collision with whatever calendar setup already exists in the target sandbox.

**Current behavior (as of 2026-07-09, current):** the per-field blank-check turned out insufficient ‚Äî an environment whose Daily Optimizer Setup already had `Base Calendar = 'BASIS'` from an older run (pre-dating the `DEMOCAL` change) kept that value forever, since blank-check only fills empties. User confirmed twice that "Create Demo Data" should fully own this singleton every run, so `CreateDailyOptimizerSetupDefault()` now UNCONDITIONALLY assigns all 5 fields (`Base Calendar := CalendarCode` ('DEMOCAL'), `Work hour Template := 'BASIS'`, `Default Skill := 'ELEKTR'`, `Order Intake Nos := 'OI'`, `Work Order Nos := 'WO'`) every run ‚Äî insert-if-missing via `if not Setup.Get() then ...`, then always overwrite, then `Setup.Modify()`. This is a deliberate exception to this codeunit's general "Ensure*-style permanent/unlogged, never touch again" pattern used elsewhere (e.g. `CreateGanttChartSetupDefaults`) ‚Äî the Daily Optimizer Setup singleton is demo-data-owned, not user-preserved.
**How to apply:** Don't reintroduce blank-checks here without re-confirming with the user ‚Äî it was already tried and explicitly rejected in favor of unconditional overwrite. For other setup/singleton seeding elsewhere in the app, default to the blank-check pattern (least surprising, preserves user customization) unless the singleton is specifically demo-data-owned like this one.

**`CreateDemoCalendar()` calendar exceptions (added 2026-07-09, revised same day to add Annual Recurring + Year-2 custom rows):** `CreateDemoCalendar()` calls `CreateDemoCalendarChanges(CalendarCode)` (right before `exit`), which populates `"Base Calendar Change"` (NOT `"Customized Calendar Change"` ‚Äî that's a different, derived/combined table used at read-time, e.g. by `ExpectedWeekDay` in codeunit 50610) via `LoadDutchPublicHolidays` + 8 direct `InsertBaseCalendarChange` calls for custom demo days-off. `InsertBaseCalendarChange(CalendarCode, ChangeDate, Desc, RecurringSystem: Integer)` is the idempotent Get-then-Insert primitive (PK is `"Base Calendar Code", "Recurring System", Date, Day` ‚Äî `Day` left at default 0/blank, only relevant for Weekly Recurring which this codeunit never uses), `LogRecord`'d so `DeleteDemoData()` wipes and recreates these every run.
- **`src/data/DutchPublicHolidays.csv`** now has a 3rd column `Recurring` (`Y`/`N`). Of the 10 Dutch holidays, only 4 are on a truly fixed calendar date every year (Nieuwjaarsdag Jan 1, Bevrijdingsdag May 5, Eerste/Tweede Kerstdag Dec 25/26) ‚Äî these appear as a SINGLE `Y` row each (one anchor date, not one row per year 2025-2030) and are inserted UNCONDITIONALLY (no `[gStartDate,gEndDate]` window filter) with `"Recurring System" = Annual Recurring`. The other 6 (Eerste/Tweede Paasdag, Hemelvaartsdag, Eerste/Tweede Pinksterdag ‚Äî all move with the lunar Paschal/Easter cycle ‚Äî and Koningsdag, which is normally fixed Apr 27 but shifts to Apr 26 in the rare year it lands on a Sunday e.g. 2025) stay `N` rows, one per year 2025-2030, window-filtered as before ‚Äî converting these to Annual Recurring would repeat the WRONG date most years, a real bug not a simplification.
- **`"Base Calendar Change"."Recurring System"` (Option) values ‚Äî do NOT guess, confirmed by decompiling the actual base-app table source** (table 7601, `src/Foundation/Calendar/BaseCalendarChange.Table.al` inside the extracted `Microsoft_Base Application_*.app`, since `al_symbolsearch` only returns `"Recurring System": Option` with no member list for base-app Option fields ‚Äî see the general decompile technique note below): `OptionMembers = " ","Annual Recurring","Weekly Recurring"` ‚Üí **0 = blank/non-recurring, 1 = Annual Recurring, 2 = Weekly Recurring** (unused here). `InsertBaseCalendarChange`'s `RecurringSystem` parameter is typed `Integer` (not the table's own Option type ‚Äî AL doesn't let an anonymous/foreign Option type cross a procedure boundary the way Enums do) and assigned directly into the Option field, which AL permits without an explicit cast; two tiny named helpers `CalRecNonRecurring(): Integer` (returns 0) / `CalRecAnnualRecurring(): Integer` (returns 1) make call sites read as intent instead of bare magic-number literals. Also confirmed: `"Base Calendar Change"`'s `OnInsert`/`OnModify`/`OnRename` triggers run `CheckEntryLine()`, which `TestField`s `Day` even for blank/non-recurring rows in a way that would reject our Day-left-blank inserts ‚Äî but this is a non-issue because `InsertBaseCalendarChange` calls bare `BaseCalendarChange.Insert()` (no `true` argument), and AL's parameterless `Insert()` defaults to `RunTrigger = false`, so `OnInsert`/`CheckEntryLine()` never fires.
- **Custom demo days-off are now 8 rows, not 4** ‚Äî the original 4 (Wk2 Tue/Wk3 Wed/Wk3 Thu/Wk5 Mon, all non-recurring, all relative to `gStartDate`) plus the same 4 patterns offset by `CalcDate('+1Y', gStartDate)` first (nested `CalcDate` calls, not a single combined `'+1Y+1W+1D'` formula, to keep each individual call as simple/proven as the ones already used elsewhere in this file) for "Year 2" of the ~2.1-year rolling window. These stay non-recurring (`CalRecNonRecurring()`) ‚Äî user explicitly confirmed neither Annual nor Weekly Recurring can express "week 2 Tuesday of whatever window this particular run computes," since they're offset from the CURRENT run's rolling `gStartDate`, not a fixed calendar month/day or weekday.

**General technique ‚Äî decompiling a base-app table when `al_symbolsearch` can't answer the question (established 2026-07-09):** `al_symbolsearch` only exposes field signatures (e.g. `"Recurring System": Option`), never Option `OptionMembers`/values, trigger bodies, or other implementation detail for base-app (non-project) objects. When that detail is actually needed (exact option ordinals, validation/trigger side effects, etc.), the on-prem symbol `.app` files under `.alpackages/` are real ZIP archives containing the FULL AL SOURCE under a `src/` folder (Microsoft ships Base Application as source, not just compiled symbols) ‚Äî extract with `unzip` (works despite a ~40-byte non-standard header prefix BC adds before the `PK\x03\x04` signature; `unzip -o` tolerates it with a warning) and `find`/`grep` the `src/` tree for the object file, e.g. `src/Foundation/Calendar/BaseCalendarChange.Table.al`. This is strictly more reliable than guessing OptionMembers/trigger behavior from the field's Caption or from general BC knowledge ‚Äî always do this before finalizing code that depends on undocumented base-app Option values or trigger side effects, per this project's "verify via symbol tools, don't guess" standing instruction. Clean up the extracted temp folder afterward (used `C:\Users\Ahmad\AppData\Local\Temp\claude\bcbaseapp_extract` this time, then `rm -rf`'d it).

**`IsWorkingDay()` now honors DEMOCAL's calendar exceptions (added 2026-07-09).** `OnRun()`'s call order was fixed so `CreateDailyOptimizerSetupDefault()` (creates DEMOCAL + its Base Calendar Change exceptions) runs BEFORE `CreateCapacityAndDayPlanning()` (was previously last, after capacity/day-planning generation, which meant DEMOCAL didn't exist yet when `IsWorkingDay()` needed it). `IsWorkingDay(DT)` still checks the Work-Hour Template weekday flags first as a fast-path/short-circuit (unchanged), then ALSO checks DEMOCAL's exceptions via `Codeunit "Calendar Management".IsNonworkingDay(DT, gCustomizedCalendarChange)`. New `EnsureBaseCalendarLoaded()` loads `gBaseCalendar`/calls `CalendarMgt.SetSource(gBaseCalendar, gCustomizedCalendarChange)` ONCE (memoized via `gBaseCalendarLoaded`, same convention as `gWorkHourTemplateLoaded`/`gResourceSkillCache`/`gResourcePoolCache`/`gNamesLoaded`) rather than per-call, because `IsWorkingDay()` runs tens of thousands of times (once per resource per candidate day across the ~2-year demo window) vs. `ExpectedWeekDay` in codeunit 50610 (which re-runs `SetSource` per call since it's called far less often ‚Äî that pattern was NOT copied here for performance reasons). Verified via `al_symbolsearch` that `Calendar Management.IsNonworkingDay(TargetDate; var CustomizedCalendarChange)` only reads/filters the already-combined record by date per call (not a cursor/stateful API), so calling `SetSource` once and reusing the same var record across many `IsNonworkingDay` calls is safe and is the standard base-app pattern. Also confirmed `gCustomizedCalendarChange` should be declared as a plain (non-`temporary`) `Record "Customized Calendar Change"` ‚Äî matches how codeunit_50610's `ExpectedWeekDay` declares its own local `CustomizedCalendarChange` variable, so this isn't a new/unproven declaration style. If `Daily Optimizer Setup` doesn't exist yet or `"Base Calendar"` is blank/invalid, `EnsureBaseCalendarLoaded()` just leaves `gCustomizedCalendarChange` unpopulated and returns ‚Äî `IsNonworkingDay()` against an empty record safely returns false (no exceptions found), so `IsWorkingDay()` never crashes in that edge case. `IsDemoWorkingDay()` (the public wrapper used by the external repair report) calls `IsWorkingDay()` internally, so it inherits this fix with no separate change needed.
**How to apply:** If any other per-date-loop procedure in this codeunit needs calendar-exception awareness, reuse `EnsureBaseCalendarLoaded()`/`gCustomizedCalendarChange` rather than adding a fresh `SetSource` call inline ‚Äî the memoization only pays off if every caller shares the same cached record.

**Work Order / Order Intake schema ‚Äî object names differ from their captions (learned 2026-07-09):** Table 50604's actual AL object name is `"Order Intake Header Opt."` (Caption = "Order Intake Header") ‚Äî searching/referencing it as `"Order Intake Header"` in code fails; always use the `Opt.`-suffixed name. Table 50608 is `"Work Order"` (file `table_50608_Workorder.al`), PK `"Work Order No."`. Enum 50612 `"Workload Source Type"`: `Manual` (0), `"Order Intake"` (1). Enum 50607 `"DayPlanning Order Intake Status"`: `Open` (0), `Ready` (1), `Released` (2), `Done` (3). **Trap:** `"Order Intake Header Opt."`'s `OnInsert` trigger unconditionally runs `"Order Date" := Today();` as its last line, regardless of what the caller set beforehand and regardless of whether the No.-auto-numbering branch fired ‚Äî so any code that wants a specific `"Order Date"` (e.g. demo data wanting a date inside a rolling window, not literally today) MUST set it via `Modify()` AFTER `Insert(true)`, never rely on the pre-Insert value sticking. `"Work Order"`'s `OnInsert` has no such trap (just audit fields `Created DateTime`/`Created By` + `SetLastModified()`). Both tables' PK fields (`"No."` / `"Work Order No."`) skip their No.-Series auto-numbering branch entirely as long as the caller sets that field to a non-blank fixed code before `Insert()` ‚Äî this is how `CreateDemoData` uses fixed `DOI*`/`DWO*` codes without touching `"Order Intake Nos"`/`"Work Order NOS"` No. Series.
